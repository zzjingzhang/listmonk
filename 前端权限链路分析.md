## 1. 问题背景

用户反馈了两个问题：

1. **CampaignAnalytics 页面**：进入页面时前端导航允许点击，API 返回 403 后全局错误提示不明确（显示原始后端消息如 `"permission denied: campaigns:get_analytics"`）
2. **某页面 loading 状态未清理**：因为 loading 状态未在错误路径清理，导致按钮一直处于禁用状态

这两个问题的根因都指向：**前端权限体系与后端权限体系之间存在断层**——前端没有完整的路由守卫、API 错误处理不区分状态码、组件缺少 `.catch()` 错误回调。

## 2. 完整链路追踪：从导航点击到 403 拒绝

### 2.1 前端 Router：无权限守卫

**文件**: `frontend/src/router/index.js`

```js
router.beforeEach((to, from, next) => {
  if (to.matched.length === 0) {
    next('/404');
  } else {
    next();
  }
});
```

路由定义中没有任何 `meta.perm` 或权限相关的元数据，`beforeEach` 守卫只处理 404 匹配，**完全没有权限校验**。这意味着任何已登录用户都可以通过手动输入 URL 直接访问 `/campaigns/analytics`，即使其角色不具备 `campaigns:get_analytics` 权限。

**问题**：用户可以直接在浏览器地址栏输入路由路径绕过导航菜单的 `v-if` 检查。

### 2.2 前端 Navigation：仅菜单级别的 v-if 隐藏

**文件**: `frontend/src/components/Navigation.vue`

```html
<b-menu-item v-if="$can('campaigns:get_analytics')" :to="{ name: 'campaignAnalytics' }" .../>
```

导航菜单使用了 `$can('campaigns:get_analytics')` 做条件渲染，如果用户无此权限，菜单项不显示。但这只是 **UI 层面的隐藏**，不是路由层面的拦截。

关键问题：`campaigns:*` 通配符组中，即使用户只有 `campaigns:get` 而没有 `campaigns:get_analytics`，整个 campaigns 组仍会展开显示（因为 `$can('campaigns:*')` 检查的是组内是否有任一权限），但 Analytics 子菜单不会出现。**然而用户仍可通过 URL 直接导航**。

### 2.3 前端 Store：无权限状态管理

**文件**: `frontend/src/store/index.js`

```js
state: {
  ...Object.keys(models).reduce((obj, cur) => ({ ...obj, [cur]: [] }), {}),
  loading: Object.keys(models).reduce((obj, cur) => ({ ...obj, [cur]: false }), {}),
}
```

Vuex Store 只管理数据模型（`lists`、`campaigns` 等）和 loading 状态，**不管理任何权限相关的状态**。用户权限信息仅通过 `main.js` 中的 `$can` / `$canList` 闭包保存在 Vue 原型上，无法被 Store 追踪或被路由守卫访问。

### 2.4 前端 $can / $canList：注册逻辑分析

**文件**: `frontend/src/main.js`

```js
Vue.prototype.$can = (...perms) => {
  // 超级管理员(id=1)直接放行
  if (profile.userRole.id === 1) {
    return true;
  }

  return perms.some((perm) => {
    // 通配符：campaigns:* 只要组内有一个权限即返回 true
    if (perm.endsWith('*')) {
      const group = `${perm.split(':')[0]}:`;
      return profile.userRole.permissions.some((p) => p.startsWith(group));
    }
    // 精确匹配
    return profile.userRole.permissions.includes(perm);
  });
};
```

**关键发现**：

- `$can` 是闭包，捕获了 `profile`（从 `/api/profile` 获取）。它无法在 Vue Router 的 `beforeEach` 守卫中被调用，因为守卫执行时 Vue 实例可能尚未创建。
- `$can('campaigns:*')` 的通配符语义是"至少有一个 `campaigns:` 前缀的权限"，而非"所有 campaigns 权限"。这导致只要用户有任何 campaign 相关权限，campaigns 菜单组就会展开。
- `$canList` 处理列表级权限，独立于 `$can`。

### 2.5 前端 Axios 拦截器：错误处理不区分 401/403/500

**文件**: `frontend/src/api/index.js`

```js
http.interceptors.response.use(
  (resp) => { /* 正常响应：清理 loading、存储数据 */ },
  (err) => {
    // 清理 loading
    if ('loading' in err.config) {
      store.commit('setLoading', { model: err.config.loading, status: false });
    }

    // 提取消息——不区分状态码
    let msg = '';
    if (err.response && err.response.data && err.response.data.message) {
      msg = err.response.data.message;  // 直接使用后端消息
    } else {
      msg = err.toString();
    }

    // 统一 toast
    if (!err.config.disableToast) {
      Toast.open({
        message: msg,
        type: 'is-danger',
        queue: false,
        position: 'is-top',
        pauseOnHover: true,
      });
    }

    return Promise.reject(err);
  }
);
```

**关键问题**：

| 状态码 | 当前行为 | 应有行为 |
|--------|---------|---------|
| 401 | 显示 toast `"invalid session"` 或类似消息 | 应清除会话并跳转至登录页 |
| 403 | 显示 toast `"permission denied: campaigns:get_analytics"` | 应显示友好的"无权限"提示，并可导航回首页 |
| 500 | 显示 toast `"Error: ..."` | 应显示"服务器错误"提示，避免暴露内部信息 |

### 2.6 后端 Middleware：认证层

**文件**: `internal/auth/auth.go` → `Middleware()`

```
请求 → Auth.Middleware → 检查 Authorization 头或 Cookie Session
  ├─ 有效 → 设置 User 到 context → 继续执行 handler
  └─ 无效 → 设置 HTTPError(403) 到 context → 继续执行 handler
```

注意：Middleware **不会直接返回 403**，而是将 `echo.NewHTTPError(http.StatusForbidden, ...)` 设置到 context 上，由后续的 handler group 中间件判断：

```go
// cmd/handlers.go 中 API group 的第二层中间件
func(next echo.HandlerFunc) echo.HandlerFunc {
    return func(c echo.Context) error {
        u := c.Get(auth.UserHTTPCtxKey)
        // 对 API 请求：如果认证失败，返回 JSON 错误
        if err, ok := u.(*echo.HTTPError); ok {
            return err  // 返回 403 JSON
        }
        return next(c)
    }
}
```

### 2.7 后端 Perm 中间件：权限校验层

**文件**: `internal/auth/auth.go` → `Perm()`

```go
func (o *Auth) Perm(next echo.HandlerFunc, perms ...string) echo.HandlerFunc {
    return func(c echo.Context) error {
        u, ok := c.Get(UserHTTPCtxKey).(User)
        if !ok {
            c.Set(UserHTTPCtxKey, echo.NewHTTPError(http.StatusForbidden, "invalid session"))
            return next(c)
        }

        // SuperAdmin(id=1) 跳过权限检查
        if u.UserRoleID == SuperAdminRoleID {
            return next(c)
        }

        // 检查用户的 PermissionsMap 中是否包含所需权限
        has := false
        for _, perm = range perms {
            if _, ok := u.PermissionsMap[perm]; ok {
                has = true
                break
            }
        }

        if !has {
            return echo.NewHTTPError(http.StatusForbidden,
                fmt.Sprintf("permission denied: %s", perm))
        }

        return next(c)
    }
}
```

**关键发现**：
- `Perm()` 接受可变参数 `perms ...string`，只要用户拥有其中**任一**权限即可通过（OR 逻辑）
- 权限检查基于 `PermissionsMap map[string]struct{}`，这是从数据库加载到内存的 map，O(1) 查找
- SuperAdmin 角色完全跳过权限检查

### 2.8 后端 Handlers：路由权限表

**文件**: `cmd/handlers.go`

与 CampaignAnalytics 相关的路由注册：

```go
// Analytics 需要精确的 campaigns:get_analytics 权限
g.GET("/api/campaigns/analytics/:type",

```

完整权限表（API 路由 → 所需权限）：

| API 路由 | 权限 |
|----------|------|
| `GET /api/campaigns` | `campaigns:get_all` 或 `campaigns:get` |
| `GET /api/campaigns/analytics/:type` | `campaigns:get_analytics` |
| `POST /api/campaigns` | `campaigns:manage_all` 或 `campaigns:manage` |
| `PUT /api/campaigns/:id/status` | `campaigns:send` |
| `GET /api/subscribers` | `subscribers:get_all` 或 `subscribers:get` |
| `GET /api/bounces` | `bounces:get` |

## 3. 问题根因分析

### 3.1 CampaignAnalytics 页面：导航可点击 + 403 提示不明

**完整链路**：

```
1. 用户点击导航 "Analytics"
   └─ Navigation.vue: v-if="$can('campaigns:get_analytics')" → 菜单项可见
      (假设用户有此权限，菜单项渲染)

2. Router 导航到 /campaigns/analytics
   └─ router/index.js: beforeEach → 无权限检查 → 直接放行

3. CampaignAnalytics.vue 组件挂载
   └─ mounted() → 调用 getCampaign(id) 和 getData()
      └─ 若用户的 campaigns:get_analytics 权限在导航渲染后被管理员撤销，
         或 profile 缓存与后端不一致，API 返回 403

4. Axios 响应拦截器
   └─ err.response.status = 403
   └─ err.response.data.message = "permission denied: campaigns:get_analytics"
   └─ Toast 显示原始后端消息 → 用户体验差

5. getData() 的 .then() 未执行（403 走 err 拦截器）
   └─ charts[typ].loading 永远为 true
   └─ 页面显示 loading 转圈，永远无法结束
```

**但即使 $can 在导航中正确工作**，仍存在绕过路径：

```
1. 用户有 campaigns:get 权限（可查看 Campaigns 列表）
2. 用户没有 campaigns:get_analytics 权限
3. 但用户直接在浏览器输入 /campaigns/analytics
4. Router 直接放行（无守卫）
5. 页面加载，API 返回 403
```

### 3.2 另一页面：loading 状态未清理导致按钮永久禁用

**文件**: `frontend/src/views/CampaignAnalytics.vue`

```js
getData(typ, camps) {
  this.charts[typ].loading = true;           // ← 设置 loading
  this.charts[typ].fn({ ... }).then((data) => {
    // ... 处理数据
    this.charts[typ].loading = false;        // ← 只在 .then() 中清理
  });
  // ⚠️ 没有 .catch() —— 如果 API 返回 403/500，loading 永远为 true
}
```

同样的问题也出现在 `queryCampaigns` 方法中：

```js
queryCampaigns(q) {
  this.isSearchLoading = true;
  this.$api.getCampaigns({ ... }).then((data) => {
    this.isSearchLoading = false;
    // ...
  });
  // ⚠️ 没有 .catch() —— isSearchLoading 永远为 true
}
```

**而 Vuex Store 的 loading 状态**则被 axios 拦截器正确清理了：

```js
// 拦截器 error 回调中
if ('loading' in err.config) {
  store.commit('setLoading', { model: err.config.loading, status: false });
}
```

但组件内部的 `isSearchLoading` 和 `charts[typ].loading` 是组件本地状态，**axios 拦截器无法清理它们**。这导致了：
- `isSearchLoading = true` → 搜索框显示 loading 状态
- `charts[typ].loading = true` → 图表区域显示 loading 遮罩
- 按钮可能通过 `:disabled="isSearchLoading"` 或类似的 loading 绑定被禁用

## 4. 前端如何根据 Server Config 权限隐藏入口

### 4.1 当前机制：Navigation.vue 的 v-if 模式

当前项目中，权限入口隐藏**仅依赖 Navigation.vue 的 `v-if`**：

| 菜单组 | 权限检查 | 子项权限检查 |
|--------|---------|------------|
| Lists | 无（始终显示） | 无 |
| Subscribers | `$can('subscribers:*')` | `subscribers:get_all`, `subscribers:get`, `subscribers:import`, `bounces:get` |
| Campaigns | `$can('campaigns:*')` | `campaigns:get`, `campaigns:manage`, `media:*`, `templates:get`, `campaigns:get_analytics` |
| Users | `$can('users:*', 'roles:*')` | `users:get`, `roles:get` |
| Settings | `$can('settings:*')` | `settings:get`, `settings:maintain` |

**权限数据来源**：`/api/profile` 返回的 `profile.userRole.permissions` 数组。

### 4.2 缺失机制：Router 导航守卫

当前路由定义中没有 `meta` 权限信息：

```js
{
  path: '/campaigns/analytics',
  name: 'campaignAnalytics',
  meta: { title: 'analytics.title', group: 'campaigns' },
  // ⚠️ 缺少 meta.perm 或类似字段
  component: () => import('../views/CampaignAnalytics.vue'),
}
```

### 4.3 缺失机制：页面组件级权限守卫

CampaignAnalytics.vue 组件内部**没有调用 `$can`** 进行权限检查，直接在 `mounted()` 中发起 API 请求。

### 4.4 完整方案：三层防护

#### 第一层：路由元信息 + 导航守卫

为每条路由添加 `meta.perm` 字段，并在 `beforeEach` 中校验：

```js
// router/index.js
const routes = [
  {
    path: '/campaigns/analytics',
    name: 'campaignAnalytics',
    meta: { title: 'analytics.title', group: 'campaigns', perm: 'campaigns:get_analytics' },
    component: () => import('../views/CampaignAnalytics.vue'),
  },
  // ...
];
```

```js
// main.js 中注册全局守卫
router.beforeEach((to, from, next) => {
  if (to.matched.length === 0) {
    next('/404');
    return;
  }

  // 权限检查
  const perm = to.meta.perm;
  if (perm && typeof Vue.prototype.$can === 'function') {
    if (!Vue.prototype.$can(perm)) {
      next('/404');  // 或跳转到专门的"无权限"页面
      return;
    }
  }

  next();
});
```

> **注意**：`$can` 是在 `initConfig()` 中注册到 `Vue.prototype` 上的，而 `initConfig()` 在路由守卫注册之后执行。因此需要确保守卫中检查 `$can` 是否已就绪，或在 `initConfig` 完成后才挂载应用。

#### 第二层：页面组件级守卫

在组件的 `beforeRouteEnter` 或 `created` 钩子中检查：

```js
// CampaignAnalytics.vue
export default {
  beforeRouteEnter(to, from, next) {
    next((vm) => {
      if (!vm.$can('campaigns:get_analytics')) {
        vm.$router.replace('/404');
      }
    });
  },
  // ...
};
```

#### 第三层：Navigation.vue 的 v-if（已存在）

这是当前已有的机制，保持不变。

## 5. 如何处理 401 / 403 / 500

### 5.1 当前处理方式

当前 axios 响应拦截器对所有错误一视同仁：

```
所有 HTTP 错误 → 提取 message → Toast.is-danger → Promise.reject(err)
```

**不区分 401（未认证）、403（无权限）、500（服务器错误）**。

### 5.2 应有的处理策略

| 状态码 | 含义 | 用户可见行为 | 技术行为 |
|--------|------|------------|---------|
| 401 | 会话过期/无效 | 提示"会话已过期，请重新登录" | 清除本地状态，跳转至登录页 |
| 403 | 无权限 | 提示"您没有执行此操作的权限" | 可选：记录无权限访问，提供返回首页按钮 |
| 404 | 资源不存在 | 提示"请求的资源不存在" | 跳转至 404 页面 |
| 500 | 服务器内部错误 | 提示"服务器错误，请稍后重试" | 不暴露内部错误详情，记录错误到监控系统 |
| 网络错误 | 连接失败 | 提示"网络连接失败" | 可选：自动重试 |

### 5.3 改进代码方案

```js
// api/index.js - 改进的响应拦截器
http.interceptors.response.use(
  (resp) => {
    if ('loading' in resp.config) {
      store.commit('setLoading', { model: resp.config.loading, status: false });
    }
    // ... 数据处理逻辑不变
  },
  (err) => {
    if ('loading' in err.config) {
      store.commit('setLoading', { model: err.config.loading, status: false });
    }

    // 根据状态码分类处理
    const status = err.response?.status;
    const serverMsg = err.response?.data?.message || '';

    if (status === 401) {
      // 会话过期 → 跳转登录
      Toast.open({
        message: i18n?.t('globals.messages.sessionExpired') || 'Session expired. Please log in again.',
        type: 'is-warning',
        queue: false,
        position: 'is-top',
        duration: 5000,
      });
      // 清除前端状态并跳转到登录页
      window.location.href = '/admin/login';
      return Promise.reject(err);
    }

    if (status === 403) {
      // 无权限 → 友好提示
      if (!err.config.disableToast) {
        Toast.open({
          message: i18n?.t('globals.messages.permissionDenied') || 'You do not have permission to perform this action.',
          type: 'is-danger',
          queue: false,
          position: 'is-top',
          duration: 5000,
        });
      }
      return Promise.reject(err);
    }

    if (status === 500) {
      // 服务器错误 → 不暴露内部信息
      if (!err.config.disableToast) {
        Toast.open({
          message: i18n?.t('globals.messages.internalError') || 'An internal error occurred. Please try again later.',
          type: 'is-danger',
          queue: false,
          position: 'is-top',
        });
      }
      return Promise.reject(err);
    }

    // 其他错误（网络错误、400 等）
    if (!err.config.disableToast) {
      Toast.open({
        message: serverMsg || err.toString(),
        type: 'is-danger',
        queue: false,
        position: 'is-top',
      });
    }

    return Promise.reject(err);
  }
);
```

## 6. 为什么后端仍是最终可信边界

### 6.1 前端权限检查的本质：UI 优化而非安全边界

前端的所有权限检查都可以被绕过：

| 前端防护 | 绕过方式 |
|---------|---------|
| Navigation.vue 的 `v-if` | 直接在浏览器输入 URL |
| Router `beforeEach` 守卫 | 在浏览器开发者工具中修改 Vue 原型上的 `$can` 函数 |
| 组件内 `v-if` 条件渲染 | 修改 DOM 移除 `display:none`，或直接调用 API |
| `disabled` 按钮 | 修改 DOM 移除 `disabled` 属性 |

**核心原因**：前端代码完全运行在用户浏览器中，用户拥有对客户端代码的完全控制权。任何前端权限检查本质上都是 **UX 优化**——避免让用户看到他们无权操作的内容，而不是真正的安全屏障。

### 6.2 后端权限检查的不可绕过性

后端权限检查链路：

```
HTTP 请求
  → Echo Router 匹配
    → Auth.Middleware（认证：验证身份）
      ├─ 失败 → 403 + "invalid session" / "invalid API credentials"
      └─ 成功 → 设置 User 到 context
        → Auth.Perm（授权：验证权限）
          ├─ 失败 → 403 + "permission denied: xxx"
          └─ 成功 → 执行 Handler 逻辑
```

**后端是不可绕过的**，因为：

1. **服务端执行环境**：Go 代码运行在服务器上，用户无法修改
2. **每次请求独立校验**：每个 API 请求都经过 `Middleware` + `Perm` 两层检查
3. **权限数据来源可信**：`PermissionsMap` 从数据库加载，不依赖客户端传入
4. **SuperAdmin 硬编码**：`SuperAdminRoleID = 1` 在代码中固定，无法伪造
5. **Session 验证**：基于数据库存储的 session，无法客户端伪造

即使攻击者绕过了所有前端检查，直接用 `curl` 调用 API，后端仍会返回 403。

### 6.3 前后端权限对照表

| 前端 $can 检查 | 后端 Perm 检查 | 一致性 |
|----------------|---------------|--------|
| `$can('subscribers:*')` | `pm(handler, "subscribers:get_all", "subscribers:get")` 等 | ⚠️ 前端通配符更宽泛 |
| `$can('campaigns:*')` | `pm(handler, "campaigns:get_all", "campaigns:get")` 等 | ⚠️ 前端通配符更宽泛 |
| `$can('campaigns:get_analytics')` | `pm(handler, "campaigns:get_analytics")` | ✅ 一致 |
| `$can('media:*')` | `pm(handler, "media:get")` / `pm(handler, "media:manage")` | ⚠️ 前端通配符更宽泛 |
| `$can('settings:*')` | `pm(handler, "settings:get")` / `pm(handler, "settings:manage")` | ⚠️ 前端通配符更宽泛 |
| 无路由守卫 | 每个 API 都有 Perm 检查 | ❌ 前端缺失 |
| 无 401 处理 | 后端返回 401/403 | ❌ 前端未区分 |

**通配符不一致**说明：前端 `$can('campaigns:*')` 表示"只要有任何 campaigns 前缀的权限就放行"，而后端对每个 API 端点要求精确权限。这是**合理的设计**——前端通配符只控制菜单组的展开/折叠，而具体操作权限由后端逐项校验。但这意味着**前端隐藏入口的粒度比后端粗**，后端仍是细粒度权限的执行者。

## 7. 修复建议总结

### 优先级 P0（影响用户体验的 Bug）

1. **CampaignAnalytics.vue 添加 `.catch()` 回调**：为 `getData()` 和 `queryCampaigns()` 添加错误回调，确保 `loading` 状态在 API 失败时被清理
2. **Axios 拦截器区分 401/403/500**：添加 `err.response.status` 判断，对 401 跳转登录页，对 403 显示友好提示，对 500 隐藏内部错误

### 优先级 P1（安全加固）

3. **Router 添加 `meta.perm` + `beforeEach` 守卫**：为每条路由定义所需权限，在导航守卫中校验 `$can(perm)`
4. **页面组件添加 `beforeRouteEnter` 权限检查**：作为路由守卫的后备

### 优先级 P2（代码质量）

5. **统一权限映射表**：将路由权限与后端 API 权限对照，确保前后端权限语义一致
6. **添加"无权限"专用页面**：当用户通过 URL 直接访问无权限页面时，展示友好提示而非 404
7. **401 全局处理与 Token 刷新**：在 axios 拦截器中检测 401，自动跳转登录页并保留原始 URL 以便登录后回跳

### 修复影响范围

| 修改文件 | 影响内容 |
|---------|---------|
| `frontend/src/api/index.js` | Axios 拦截器增加 401/403/500 分支 |
| `frontend/src/router/index.js` | 路由定义增加 `meta.perm` 字段 |
| `frontend/src/main.js` | `beforeEach` 守卫增加权限校验 |
| `frontend/src/views/CampaignAnalytics.vue` | 添加 `.catch()` 清理 loading |
| `frontend/src/views/*.vue`（可选） | 各页面添加 `beforeRouteEnter` 权限检查 |
