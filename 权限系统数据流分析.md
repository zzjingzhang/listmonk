# 权限系统数据流分析与回归验证方案

## 1. 问题概述

管理员修改角色权限并禁用API用户后，出现两个问题：
1. **旧API token仍能访问部分接口一段时间** —— 缓存一致性问题
2. **前端权限菜单显示与后端拒绝不一致** —— 前后端权限数据源不同步

---

### 3.4 缓存机制：cacheUsers + auth.CacheAPIUsers

**核心文件**：
- 缓存函数：[cmd/users.go#L352-L375](file:///Users/zhangjing/Desktop/so-coders/0508-und-p/listmonk/cmd/users.go#L352-L375)
- Auth缓存：[internal/auth/auth.go#L115-L146](file:///Users/zhangjing/Desktop/so-coders/0508-und-p/listmonk/internal/auth/auth.go#L115-L146)

#### cacheUsers 实现：

```go
func cacheUsers(co *core.Core, a *auth.Auth) (bool, error) {
    users, err := co.GetUsers()  // 从DB读取所有用户
    
    hasUser := false
    apiUsers := make([]auth.User, 0, len(users))
    for _, u := range users {
        // 🔴 关键过滤条件：只缓存 type=api 且 status=enabled 的用户
        if u.Type == auth.UserTypeAPI && u.Status == auth.UserStatusEnabled {
            apiUsers = append(apiUsers, u)
        }
        if u.Type == auth.UserTypeUser {
            hasUser = true
        }
    }
    
    a.CacheAPIUsers(apiUsers)  // 全量替换缓存
    return hasUser, nil
}
```

#### auth.CacheAPIUsers 实现：

```go
func (o *Auth) CacheAPIUsers(users []User) {
    o.Lock()
    defer o.Unlock()
    
    o.apiUsers = map[string]User{}  // 🔴 先清空再重建
    for _, u := range users {
        o.apiUsers[u.Username] = u
    }
}
```

#### API token 认证流程：

```
API请求 (Authorization: token <key>:<secret>)
        ↓
auth.Middleware [internal/auth/auth.go#L286-L333]
        ↓
1. 解析 Authorization 头 → (key, token)
        ↓
2. GetAPIToken(key, token) 从缓存中查找
   [internal/auth/auth.go#L136-L146]
   - 读锁保护下读取 o.apiUsers[key]
   - 使用 subtle.ConstantTimeCompare 安全比较token
        ↓
3. 认证成功 → User 对象存入 context
   认证失败 → 403 Forbidden
        ↓
4. auth.Perm 中间件检查权限
   [internal/auth/auth.go#L336-L367]
   - 超级管理员 (role_id=1) 直接放行
   - 检查 PermissionsMap 中是否包含所需权限
```

**核心文件**：[internal/auth/auth.go#L395-L423](file:///Users/zhangjing/Desktop/so-coders/0508-und-p/listmonk/internal/auth/auth.go#L395-L423)

## 4. 问题根源深度分析

### 4.1 问题1：旧API token仍能访问部分接口

**根本原因**：**API用户权限缓存在内存中，更新路径不完整**

```
时序图：
T0: API用户创建，token=abc，权限=[lists:get_all, subscribers:get]
    → cacheUsers() 被调用，auth.apiUsers 缓存该用户

T1: 管理员修改角色，移除 subscribers:get 权限
    → UpdateUserRole() 被调用
    → ✅ cacheUsers() 被调用，缓存更新（权限变化）
    → 新权限即时生效

T2: 管理员禁用该API用户 (status=disabled)
    → UpdateUser() 被调用
    → ✅ cacheUsers() 被调用
    → 过滤条件 u.Status == enabled 生效，用户从缓存移除
    → 新请求被拒绝 ✅

T3: 管理员删除该API用户
    → DeleteUser() 被调用
    → ✅ cacheUsers() 被调用
    → 用户从缓存移除
    → 新请求被拒绝 ✅

T4: 管理员修改角色权限，但该角色关联多个API用户
    → UpdateUserRole() 被调用
    → ✅ cacheUsers() 被调用，所有API用户缓存刷新
    → 权限变更即时生效 ✅
```

**潜在延迟场景**：
1. **多实例部署**：如果运行多个listmonk实例，cacheUsers只刷新当前实例缓存，其他实例需等待重启或下一次刷新
2. **竞态条件**：cacheUsers执行期间（读DB→写缓存），刚好有请求进来使用旧缓存
3. **CreateRole未刷新**：虽然影响小，但新角色分配给用户后需要UpdateUser才会刷新

---

### 4.2 问题2：前端权限菜单与后端拒绝不一致

**根本原因**：**前端权限基于登录时的profile快照，后端权限实时从DB读取**

```
时序对比：
T0: 用户登录，前端调用 /api/profile 获取权限快照
    权限=[campaigns:get, campaigns:manage]
    → 前端 $can('campaigns:manage') = true
    → 菜单显示 "新建活动" ✅

T1: 管理员修改该用户角色，移除 campaigns:manage 权限
    → UpdateUserRole() → cacheUsers() 刷新API缓存
    → 后端会话用户权限：每次请求从DB读取，已无该权限 ❌

T2: 用户在前端点击 "新建活动"
    → 前端菜单仍显示（因为profile未刷新）✅
    → 发送 POST /api/campaigns 请求
    → 后端 validateSession → 从DB读取最新权限
    → auth.Perm 检查：无 campaigns:manage 权限
    → 返回 403 Forbidden ❌

T3: 用户刷新浏览器（F5）
    → initConfig() 重新调用 /api/profile
    → 获取最新权限，菜单不再显示 ✅
```

**代码证据**：
- 前端profile加载仅在 [initConfig()](file:///Users/zhangjing/Desktop/so-coders/0508-und-p/listmonk/frontend/src/main.js#L37) 调用一次
- 前端 `$can()` 使用的是 [profile.userRole.permissions](file:///Users/zhangjing/Desktop/so-coders/0508-und-p/listmonk/frontend/src/main.js#L56-L63) 静态数组
- 后端每次请求都调用 [co.GetUser()](file:///Users/zhangjing/Desktop/so-coders/0508-und-p/listmonk/internal/auth/auth.go#L417) 从DB读取

## 7. 总结

| 组件 | 数据源 | 更新时机 | 实时性 |
|------|--------|----------|--------|
| API用户认证 | `auth.apiUsers` 内存缓存 | 用户/角色变更时调用 `cacheUsers()` | 单实例即时，多实例延迟 |
| 会话用户权限 | DB实时查询 | 每次请求 | 完全实时 |
| 前端菜单显示 | 登录时 `profile` 快照 | 页面刷新/重新登录 | 仅刷新后同步 |
| 前端权限检查 | `profile.userRole.permissions` | 页面刷新/重新登录 | 仅刷新后同步 |

**核心结论**：
- API token撤销在单实例下是即时的，多实例需要重启同步
- 后端权限检查对会话用户是完全实时的
- 前端菜单显示依赖登录时的权限快照，存在不一致窗口
- 三类回归验证覆盖了主要场景，可作为自动化测试用例
